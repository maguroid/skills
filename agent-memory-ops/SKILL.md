# agent-memory-ops — ハブ型エージェントメモリの運用

## アーキテクチャ

エージェントの学習事項・知見を、特定ハーネスの機能に閉じ込めず、ハブワークスペース
（オーケストレーションの起点となるリポジトリ）が持つ**ただのMarkdownディレクトリ**
`agent-memory/` を正本ストアとして運用する。

- **実体はハブリポジトリ内**: `agent-memory/` はどのハーネス・どのツールからも
  読み書きできる平文の知識ストア。特定ハーネスへの接続は後述のアダプタが担う。
- **同期はハブの既存インフラに乗せる**: ハブリポジトリの自動コミット・プッシュ・プル
  （フック等）がそのままメモリを端末間同期する。専用の同期機構は作らない。
- **ハブ境界＝信頼境界**: ハブ（個人ワークスペース・チームワークスペース・クライアント
  ハブ）ごとにメモリは分離され、自動では跨がない。ハブを跨ぐ知識の移動は、月次GCの
  昇格判定という**意図的なゲート**だけを通す（無審査の自動同期にしない）。

## データ規約

- 1ファイル1知識。ファイル名は kebab-case（ハイフン区切り）。
- frontmatter は `name`（slug）と `description`（一文サマリ）を必須とする。
  書き込んだハーネスが固有の `metadata`（Claude Code の `node_type` /
  `originSessionId` 等）を付けるのは許容する。
- 索引 `MEMORY.md` に1知識1行（`- [タイトル](ファイル名) — 要点`）を保つ。
  ディレクトリ内に `index.md` は置かない。
- ハブ側の文書規約（OKF 等）は**適用外**とし、ハブのリンタがあれば
  `agent-memory/` を除外設定に入れる。
- **保護対象**: 他のスキル・ルールが名指しで参照するメモリ（例: ブリーフ正本ストアの
  所在解決記録 `creative-brief-stores`）は、GCで統合・削除しない。

## ハーネス接続（アダプタ）

ストア本体は中立に保ち、各ハーネスの「自動で読む・書く」への接続だけをアダプタとして足す。

- **Claude Code**: `~/.claude/projects/<プロジェクトディレクトリ名>/memory` から
  `agent-memory/` へのシンボリックリンクで、組み込み自動メモリ機能（セッション開始時の
  自動注入・会話からの自動書き込み）にそのまま接続する。プロジェクトディレクトリ名は
  ハブ絶対パスの `/` と `.` を `-` に置換したもの。新端末への配布は dotfiles（chezmoi）の
  `run_once_after_link-claude-agent-memory.sh` が担い、ハブのリスト（`HUBS`）を
  ループして冪等にリンクを張る（ハブ追加＝リスト1行追記。run_once は内容変更で
  全端末再実行される）。
- **その他のハーネス（codex 等）**: 自動注入機構がないため、ハブの AGENTS.md に
  「**作業開始時に `agent-memory/MEMORY.md` を読み、関連するメモリ本文を参照する**」
  旨を明記して接続する。書き込みは上記データ規約（frontmatter＋索引1行）に従って
  ファイルを追加・編集すればよい。

## 月次GC手順

ハブのタスク管理に月次タスクとして登録し、期限が来たら実行する。

1. **重複・陳腐化の整理**: 内容が重複するメモリを統合し、事実が変わって古くなった
   メモリを削除・修正する。
2. **命名の正規化**: kebab-case に揃え、リネーム時は参照箇所（`MEMORY.md`・
   wikilink）も更新する。
3. **索引整合**: `MEMORY.md` の行と実ファイルの過不足を機械的に突き合わせる
   （`diff <(ls) <(索引から抽出)` 等）。
4. **正本への圧縮**: ハブ内の正本ドキュメント（`ideas/`・wiki 等）と内容が重複する
   メモリは、正本への参照1行＋要点に圧縮する。複製を残さない。
5. **ハブ跨ぎ知識の昇格判定**: `feedback-assetization` スキルが定義する
   「暗黙の受け入れ基準の資産化」の型（一次データ→抽出→階層判定→焼き付け）を、
   エージェントの知識一般に適用する工程。メモリを知識の最下層の一次データとみなし、
   各メモリに昇格テスト「**このハブの外のセッションでも生きる知識か**」を問い、
   生きるものは性質で昇格先を選ぶ:
   - 全セッションで常時従うべき運用ルール → **グローバル CLAUDE.md**（chezmoi 配布）。
   - 特定タスク種別の手順・ノウハウ → **該当するグローバルスキル**
     （既存スキルへの追記を新設より優先）。
   - 主体・チーム固有の内容知識 → **主体の正本ストア**（チーム wiki・`ideas/` 等）。

   昇格の作法は型と共通:
   - 昇格する知識には**根拠となった実例**（どの案件・どのセッションの何に由来するか）を添える。
   - 由来した回数（n）を正直に保つ。迷うもの（n=1 で再現未確認など）は昇格させず残し、
     次回のGCで再判定する（**2回目に出会った知識が昇格の有力候補**）。
   - 昇格済みの知識に**反例**が出たら、削除ではなく適用条件を絞るか元の階層へ降格する。
   - 昇格後のメモリは項目4と同様に正本への参照へ圧縮する（正本と複製の一方向規約）。
   - 共有領域（チーム wiki 等）へ昇格させる際は機密境界を確認する
     （個人や他主体の情報を持ち込まない）。

完了したらハブのタスクを完了にし、翌月分の期限で再登録する。

## 新しいハブへの導入手順

対象ハブのリポジトリで、以下を順に行う（全体で冪等になるように）。

1. **ストアの用意**: `<ハブ>/agent-memory/` を作る。既存のClaude Codeメモリがある
   ハブでは `mv ~/.claude/projects/<プロジェクトディレクトリ名>/memory <ハブ>/agent-memory`
   で実体を移設する。メモリがまだ無ければ空ディレクトリと空の `MEMORY.md` で開始してよい。
2. **Claude Code アダプタ**: 旧パスから `ln -s <ハブ>/agent-memory ~/.claude/projects/<プロジェクトディレクトリ名>/memory`。
   あわせて dotfiles（chezmoi ソース）の `run_once_after_link-claude-agent-memory.sh` の
   `HUBS` リストにハブのパスを1行追加し、コミット・プッシュする（chezmoi の Edit Flow に従う）。
3. **他ハーネスアダプタ**: ハブの AGENTS.md に「作業開始時に `agent-memory/MEMORY.md`
   を読む」指示を含める（次項のポインタ節に含めてよい）。
4. **リンタ除外**: ハブに文書規約チェック（OKF 等）があれば `agent-memory/` を除外に追加。
5. **ハブ文書へのポインタ追記**: ハブの AGENTS.md（または CLAUDE.md）に薄い節を1つ追加する。
   書くのは「実体の場所と接続方式」「同期がハブの既存フックに乗ること」「運用の定義元は
   このスキル（`agent-memory-ops`）であること」「ハブ固有の除外設定」のみ。
   **GC手順や規約の本文をハブ側に複製しない**（ドリフトの温床になる）。
6. **索引更新**: ハブにルート索引（`index.md` 等）があれば `agent-memory/` の一行を追加。
7. **月次GCタスクの登録**: ハブのタスク管理システムに月次GCタスクを追加する
   （手順参照先はこのスキル）。
8. **検証**: リンクの解決、ハブのコミット・プッシュでメモリが同期されること、
   リンタが通ることを確認する。

## トラブルシュート

- 新端末でメモリが見えない → ハブリポジトリのクローンと `chezmoi apply`（run_once）の
  実行順を確認。スクリプトはハブ未クローンでもリンクだけ先に張る（クローン後に有効化）。
- リンク先に実体ディレクトリが既にある → run_once スクリプトは自動マージせず
  タイムスタンプ付きで退避する。退避分は手動でマージする。
- ハブのリポジトリ改名・移動 → Claude Code のプロジェクトディレクトリ名が変わるため、
  旧パスのメモリが孤児化する。旧 `~/.claude/projects/<旧名>/memory` を確認し、
  必要分を移してから削除する。
