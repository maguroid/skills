# feedback-assetization — 暗黙の受け入れ基準の資産化と記憶システム

## 原則（1分版）

テスト・仕様で正解を機械判定できる仕事はエージェントが自走できるが、正解が人間の
頭の中の暗黙判断にしかない仕事は、検証ループが毎回人間を経由する。このとき大量に
発生するフィードバックの実体は「修正指示」ではなく**遅延した要件定義**である。
対策はフィードバックを消費して捨てず、一次データとして恒久資産（ブリーフ・受け入れ
基準・テスト）へ焼き付けること。ゴールはフィードバックゼロではなく**ラウンド数の削減**。

適用対象は制作物に限らない。コードの品質基準・レビュー観点、運用手順の良し悪し、
調査の深さと形式、事業判断——**受け入れ基準が暗黙なあらゆる作業**にこの型を適用する。

## 記憶システム（基盤）

型の運搬基盤として、エージェント中立の「ハブ型」メモリを使う。

- **実体**: ハブワークスペース（オーケストレーションの起点リポジトリ）内の
  `agent-memory/` ディレクトリ。1ファイル1知識のMarkdown＋索引 `MEMORY.md`。
  どのハーネス・ツールからも読み書きできる平文ストア。
- **同期**: ハブリポジトリのgit（既存のコミット・プッシュ・プルのフック）に乗せる。
  専用の同期機構は作らない。
- **ハーネス接続**（2026-07-05 改訂: ハーネスのネイティブメモリから分離）:
  - **読み込み**: SessionStart フックで機械注入する。共通スクリプト
    `scripts/memory_inject.sh`（このスキル同梱）が、cwd がハブ配下のとき
    `agent-memory/MEMORY.md` を標準出力へ出し、Claude Code / codex の
    SessionStart フック仕様によりコンテキストへ注入される。**ハブでのセッションでは
    索引が開始時からコンテキストにある**——メモリを「探しに行く」必要はない。
    フック未対応のハーネスは AGENTS.md の「作業開始時に `agent-memory/MEMORY.md`
    を読む」指示で代替する。
  - **書き込み**: 明示的に行う（下記データ規約）。
  - **ネイティブメモリとの分離**: ハーネス組み込みの自動メモリ（Claude Code の
    auto-memory / auto-dream 等）とは**ストアを共有しない**（シンボリックリンク接続は
    廃止）。ネイティブ層はハーネス固有の使い捨て第1層として自由に動作させ、恒久知識は
    dreaming（定期GC工程4）とレビューを通じてハブ層へ昇格する。共有すると、ハーネスの
    自動整理機構がハブ正本を人間ゲートなしで書き換える経路になるため。
- **データ規約**: frontmatter は `name` と `description` を必須（ハーネス固有の
  `metadata` は許容）。索引 `MEMORY.md` に1知識1行。ファイル名は kebab-case。
  `agent-memory/archive/` サブディレクトリは昇格済み実例記録の保管庫で、索引には
  載せない（定期GCの工程7が定義元）。ハブの文書規約（OKF等）は適用外で、
  リンタからは除外する。
- **ハブ境界＝信頼境界**: メモリは自動ではハブを跨がない。ハブ跨ぎの知識移動は
  定期GCの昇格判定という意図的なゲートだけを通す。
- **ハブレジストリ（複数ハブ運用時）**: グローバル層に、ハブごとの所在・主体・
  扱う知識の範囲・機密レベルを列挙したレジストリを1ファイル置ける（既定の置き場所は
  `$HOME/.agents/hubs.md`。dotfiles 等の配布機構があればそれに載せる）。レジストリは
  **所在とスコープのメタデータ層（読み取りゲート）であり、知識の共有路ではない**——
  知識の実体を書かず、引くのは①正本の解決がハブ内で失敗したとき②定期GCの昇格判定で
  ハブ外の昇格先候補を列挙するとき③ユーザーが明示的に他ハブの参照を指示したとき
  ④dreaming の日次自動化（定期GC工程4）が作業フォルダスコープ→ハブの対応を解決する
  とき、の4場面に限る。レジストリにはハブごとの**作業フォルダスコープ**（どのフォルダ群の
  セッションがどのハブに帰属するか。どのスコープにも属さないセッションを引き受ける
  既定ハブを1つ含む）も宣言できる。全ハブの索引を自動注入したり暗黙に横断検索したりしない（どこからでも
  読める状態は昇格圧力を消し、知識が正しい階層に焼き付かなくなる）。ハブを跨いで
  読んだ知識が2回目も必要になったら、その場で複製せず昇格判定へ回す。

## 階層モデル

抽出した原則は「どの階層まで一般化できるか」で焼き付け先を分ける。

| 階層 | 範囲 | 置き場所 |
|---|---|---|
| L2 主体 | 人・会社の価値観。その主体の全成果物と事業判断に適用 | 主体の L1/L2 ストア |
| L1 ドメイン | 活動種別（サイト制作・コピー等）の原則。同ドメインの別案件へ転用可 | 主体の L1/L2 ストア |
| L0 案件 | この案件固有の決定。転用しない | 案件リポジトリのブリーフ（`docs/design-brief.md` 等） |

- **昇格テスト**: 「案件（またはドメイン）が変わってもこの原則は生きるか？」——生きるなら1つ上の階層へ。
- **正本と複製**: 正本は各階層に1箇所。案件側ブリーフが自己完結を要する場合はL1/L2を複製してよいが、複製側冒頭に正本の所在を明記し、改訂は必ず正本→複製の一方向。複製側だけを直したくなったら正本を先に直すサイン。
- **ストアは平文で書く**: L0/L1/L2・主体・昇格・複製規約などの**管理語彙をコンテンツ文書（ストア・ブリーフ）の本文に持ち込まない**。ストアは持ち主が半年後にそのまま読める言葉で書く。階層対応・主体境界・分化条件などの管理情報は、ハブのメモリ記録（所在解決の正本記録）とこのスキルが持つ。案件ブリーフに階層タグを付ける場合も各節一行にとどめる。管理語彙が変わるたびにコンテンツ文書を書き換える羽目になったら、それは語彙が侵食しているサイン。
- **所有の一致**: ストアは**その主体が所有するインフラ**に置く（個人→個人ワークスペース、会社・チーム→その組織のリポジトリ・wiki、クライアント→クライアント自身の基盤）。抽出した原則は主体の資産であり、支援者・オーケストレーター側のワークスペースに正本を置かない。ハブが担うのは所在記録と運搬のみ。
- **主体境界は実態で引く**: 主体の単位は登記や名義ではなく、実態（関与者・資産の帰属）で決める。一人法人のように人と組織が未分化なら**合併主体**（例:「個人/会社」）が正しい単位であり、これは所有原則の例外ではない（インフラの所有者は同一人）。**分化イベント**——本人以外の関与者の発生、資産帰属を分ける必要の発生——が起きた時点で主体を分割し、原則を実態に沿って仕分けてそれぞれの所有インフラへ移す。分化前に境界を推測で引かない（n不足の分類は誤分類になる）。分化イベントはストア自身の位置づけ節に明記し、ストアを使う（＝委譲時に読む）たびに再評価される状態を保つ。
- 反例が出た原則は削除せず、適用条件を絞るか下位階層へ降格する。原則には根拠となった実例（どの案件のどの判断か）を添え、由来した案件数（n）を正直に保つ。

この階層と昇格テストはブリーフに限らず**エージェントの知識一般**に同型で適用できる
（定期GCの昇格判定がその適用例）。新しく「知識を恒久資産へ焼き付ける仕組み」を
設計するときは、この型を再発明せず参照・適用すること。

## 正本の解決

ブリーフ正本の所在はこの公開スキルにハードコードしない（個人・組織のパスを含むため）。
**所在の正本記録はハブのエージェントメモリにある**。次の順で解決する。

1. **主体を特定する**: この成果物の受け入れ基準を最終的に持つのは誰か（ユーザー個人か、
   どの会社・組織か）。案件のリポジトリや依頼の文脈から判断し、迷ったらユーザーに確認する。
2. **コンテキスト内のメモリ索引を見る**: ハブセッションでは `MEMORY.md` 索引が
   SessionStart フックにより**すでに注入されている**。ファイルシステムの探索から始めず、
   まず注入済みの索引から所在記録（例: `creative-brief-stores`）を探し、あればそのメモリ
   本文を読む。注入がないハーネスではハブの `agent-memory/MEMORY.md` を読む。
3. **案件側の複製から正本を辿る**: 案件リポジトリに L0 ブリーフや L1/L2 の複製があれば、
   複製規約によりその冒頭に正本の所在が明記されている。
4. **ハブレジストリを引く**: ハブ内で解決できず、複数ハブ運用のレジストリ（上記
   「記憶システム」参照）があれば、主体に合致するハブをそこから探す。他ハブを参照する
   際はエントリの機密レベルに従い、現在の文脈（コミット・委譲プロンプト等）への
   持ち込み可否を確認する。
5. どれでも解決しなければユーザーに所在を確認する。主体にまだストアがなければ、
   **主体が所有するインフラ**（個人→個人ワークスペース、会社・チーム→その組織の
   リポジトリ・wiki、クライアント→クライアント自身の基盤）への新設を提案し、
   作成後に所在をハブのエージェントメモリへ記録する。
6. L0（案件ブリーフ）はメモリを介さず、常に案件リポジトリ側に置く。

## 場面別の手順

### A. 案件・取り組みの開始時（事前採掘）

1. 主体を特定して L1/L2 ストアを解決し、この案件に関わる原則を作業コンテキストへ取り込む。
2. 暗黙基準の先出し: 形容詞（「洗練された」等）で語らせず、**参照例**（好きな実例・避けたい実例）をユーザーから集める。
3. 判断は「選ぶ」方が「語る」より一桁安い——**複数バリアント提示→選択**を既定の進め方にする。
4. L0 ブリーフの置き場所（例: `docs/design-brief.md`）と、フィードバック記録の規約（下記C）を最初に決めて案件のAGENTS.md等に書いておく。

#### 既存成果物のデザイン比較は隔離プレビューで行う

既存のサイト、アプリ画面、文書、スライドなどに対する複数案を、選定前に実物へ
混ぜ込まない。汎用の可視化画面だけで代用せず、**元の成果物の文脈を再現した一時的な
隔離プレビュー**を使う。

1. **実物の規約を先に読む**: 案件のデザインシステム、トークン、フォント、余白、
   コンポーネント、実画像、確定コピー、レスポンシブ規則を読み込む。プレビュー側の
   デフォルトテーマや、別ツールのデザインシステムで上書きしない。
2. **作業フォルダの外へ置く**: 選定前の比較物はOSの一時領域などに置き、既存成果物の
   リポジトリを汚さない。必要なときだけ別のローカルポートで配信し、実サイト・実アプリは
   変更しない。公開用資産を参照する場合は、その参照元サーバーへの依存を明示する。
3. **比較変数を一つに絞る**: 本文、画像、表示幅、周辺レイアウトなど判断対象外の条件を
   全案で揃え、選びたい差だけを変える。案名はA/B/C等の中立ラベルにし、優劣を先に
   誘導しない。
4. **比較UIを成果物と分離する**: 切り替えメニューは「比較用・実装外」と明記する。
   成果物本体のデザイン規約に従いつつ、比較操作が本番UIの一部に見えないようにする。
5. **実表示を検証する**: 代表的なデスクトップ幅とモバイル幅で、資産の読み込み、
   横スクロール、操作、コンソールエラーを確認する。見た目だけでなく、選択状態や
   キーボード操作も確認する。
6. **選定後にだけ適用する**: ユーザーが案を選んだら、その判断理由を案件の設計文書へ
   記録して選択案だけを実装する。比較サーバーを停止し、一時ファイルを削除する。

新規の独立した図解・チャート・シミュレーターは通常の可視化手段でよい。隔離プレビューを
使うのは、**既存成果物のデザイン文脈そのものが評価条件になる比較**である。

> 例: 2026-07-10、既存コーポレートサイトのプロフィール組版比較を汎用テーマの
> プレビューで提示したところ、案件のデザインガイドラインから外れていると指摘された。
> 作業フォルダ外の一時HTMLを別ポートで配信し、実サイトの色、組版、写真、全文、
> レスポンシブ規則を再現して、切り替える組版だけを変えた比較へ修正した。

### B. タスクの委譲時（同梱チェック）

委譲プロンプトに次を含めたか確認してから送る:

- 主体の L1/L2 ストアの該当原則（またはファイルを読む指示。委譲先がストアを読めない場合は本文へ転記）。
- 案件にL0ブリーフがあればその参照。
- 「ユーザーのフィードバックを受けたら、判断理由をdocsに逐次注記する」指示（下記C）。

### C. 作業中（フィードバックの記録）

ユーザーのフィードバックで判断が変わるたび、案件の設計ドキュメントに
**「(日付 ユーザー指摘/判断) before → after ＋理由」** を逐次注記する。
git のコミットメッセージにも何を差し替えたかが残る粒度でコミットする。
この記録が後の抽出の一次データになる——トランスクリプトは消えるが docs と git 履歴は残る。

### D. 抽出（ブリーフを起こす・更新する）

フェーズ完了時、フィードバック多発の後、またはユーザーの依頼で実行する。

1. **ソース収集**: 案件の git 履歴（差し替え系コミットのdiff）・設計docsの判断注記・PR/Issueコメント・残っていればセッショントランスクリプト。長大なログの調査は、ハーネスがサブエージェントを持つならそちらへ委譲する。
2. **インベントリ化**: 各フィードバックを before/after・カテゴリ・背後の好み（推測は推測と明示）で一覧化する。
3. **パターン要約**: 繰り返し現れる好みの軸を5〜10個に要約する。頻出ほど重要。
4. **階層判定**: 各パターンに昇格テストを適用し、L0/L1/L2 に仕分ける。
5. **焼き付け**: L0→案件ブリーフへ、L1/L2→主体のストアへ追記（実例つき）。案件側に複製がある場合は正本→複製の順で更新する。機械判定できる基準（レイアウト崩れ・コンソールエラー・受け入れ条件等）はブリーフでなくテスト・ハーネス側へ落とす。
6. **不足の明示**: ブリーフ化に足りない領域（ユーザーに聞くべきこと）を成果物の末尾に列挙する。

### E. 減衰測定

次のイテレーションではブリーフを同梱してから着手し、フィードバックのラウンド数が
前回より減ったかを観察する。減っていなければ、ブリーフの記述が抽象的すぎる
（実例が足りない）か、階層判定が誤っている可能性を疑う。

## 定期GC（メモリの衛生と昇格）

実行時期の既定は**ストア自己記述方式**: `MEMORY.md` 冒頭の「最終GC: YYYY-MM-DD」行が
30日を超えているのに気づいたセッションが、作業の区切りでGCを提案する。外部の
タスクシステム・スケジューラは前提にしない（あれば併用可）。

1. **重複・陳腐化の整理**: 重複メモリの統合、古くなったメモリの削除・修正。
2. **命名の正規化**: kebab-case に揃え、リネーム時は参照箇所も更新。
3. **索引整合**: `MEMORY.md` と実ファイルの過不足を機械的に突き合わせる。
4. **トランスクリプト・マイニング（dreaming パス）**: メモリに書かれた知識だけでなく、
   **書き残されなかった反復パターン**をセッショントランスクリプトから採掘する工程
   （Claude Managed Agents の Dreams を参考に 2026-07-05 導入）。
   - **入力**: このハブの前回マージ（下記マーカー）以降のセッショントランスクリプト
     （Claude Code なら `~/.claude/projects/<cwdのエンコード名>/` 配下。他ハーネスは
     各自のログ置き場）。
   - **採掘観点**（合成方針。Dreams の instructions に相当）: ①ユーザーの修正・指摘で
     判断が変わった箇所（before/after）②複数セッションで繰り返された質問・ミス・
     回避策 ③収束しつつあるワークフロー。一回きりのデバッグ経緯・案件固有の詳細は
     拾わない。既にメモリ・ブリーフ・ルール・スキルに焼き付いている内容は除外し、
     **未記載の差分だけ**を候補にする。
   - **非破壊＋レビュー**: 候補（メモリファイル＋索引行）はハブリポジトリの
     `dreaming/<日付>` ブランチにコミット・プッシュし、本流へのマージは常に人間の
     レビューを通す（エージェント支援で候補ごとに採用/破棄を確認してから統合し、
     索引整合まで行う）。自動マージ・本流への自動書き込みはしない。
   - **日次自動化**: 会話量の多い運用では、夜間ジョブが候補生成〜ブランチのプッシュ
     までを毎日行う。ジョブは全ハブ共通の1本とし、ハブレジストリの作業フォルダ
     スコープ宣言に従って**ハブごとに分けて**採掘する（1回の採掘に複数ハブの
     セッションを混ぜない。どのスコープにも属さない作業フォルダのセッションは
     既定ハブへ）。
   - **supersede**: 未マージの dreaming ブランチが残ったまま次回実行が来たら、
     旧ブランチを破棄し「最後のマージ時点から今回まで」の全期間で採掘し直す
     （open ブランチはハブごとに常に最大1本。複数期間で反復したパターンは n が
     増えて候補として強くなる）。採掘窓のマーカーはブランチ内のファイルで持ち、
     マージ時にのみ本流で前進する。
   - **nの扱い**: 複数セッションに現れたパターンが有力候補（n≥2）。単発（n=1）は
     候補に載せてもメモリ化は保留し、次回再判定（昇格判定と同じ思想）。
   - **手動の単体実行**: GCと独立に「トランスクリプトから学びを抽出して」等の依頼で
     実行してよい。その場合メインオーケストレーターは自分で読まず、サブエージェントへ
     パス・採掘観点・報告フォーマット（候補一覧のみ）を渡して委譲する（再委譲不可を明記）。
5. **正本への圧縮**: ハブ内の正本ドキュメントと内容が重複するメモリは、正本への参照へ圧縮。
6. **ハブ跨ぎ知識の昇格判定**: 上記の階層モデルを知識一般に適用する工程。各メモリに
   昇格テスト「**このハブの外のセッションでも生きる知識か**」を問い、生きるものは
   性質で昇格先を選ぶ——常時従うべき運用ルールは**グローバル CLAUDE.md**へ、
   特定タスク種別の手順は**該当グローバルスキル**へ（新設より追記優先）、主体・チーム
   固有の内容知識は**主体の正本ストア**へ。昇格の作法（実例添付・nの管理・反例は降格・
   一方向の複製規約）は階層モデルと共通。迷うもの（n=1）は残して次回再判定
   （2回目に出会った知識が有力候補。他ハブへ読みに行った知識の再需要も同じ扱い）。
   昇格先候補の列挙には、あればハブレジストリを使う。共有領域へは機密境界を確認
   （レジストリの機密レベルが判断材料になる）。
7. **昇格済み実例記録のアーカイブ**: 昇格が完了したメモリは、正本に無い独自情報
   （追加の運用知見・技術詳細・所在ポインタ）が残っているかで扱いを分ける。
   残っていないもの（正本の由来を示すだけの純粋な実例記録）は `agent-memory/archive/`
   へ移動し、`MEMORY.md` の索引から行を外す——索引は毎セッション注入される
   コンテキスト費用なので、live には現役の知識だけを載せる。移動時はファイル名を
   変えず、本文に正本の所在が明記されていることを確認する。live 側から移動対象への
   `[[name]]` リンクが正本参照で置き換えられるかも確認する。独自情報が残るものは
   live に留め、正本と重複する部分だけ正本への参照へ圧縮する。アーカイブは削除では
   ない——反例が出て原則の適用条件を絞る・降格する際の一次データ（実例・n）として
   `archive/` を参照する。索引の末尾にアーカイブの所在を示す一行を置く（1行のみ、
   個別ファイルは列挙しない）。

**保護対象**: 他のスキル・ルールが名指しで参照するメモリ（例: 所在解決の正本記録
`creative-brief-stores`）はGCで統合・削除しない。完了したら「最終GC」日付を更新する。

## 診断への使い方

「マイクロマネジメントになっている」という相談が出たら: その仕事の
①判定の外部化度（機械判定できる正解があるか）と②仕様の圧縮率（言葉で伝えきれるか）
を確認する。両方低ければそれは仕事の性質であり、処方はこのスキルのサイクル
（記録→抽出→焼き付け→同梱）の導入。エージェントやユーザーの側の欠陥として扱わない。

## セットアップ・導入・トラブルシュート

新しい環境への導入（常時層の設置・ハブと同期フックの新設・ハーネスアダプタ・
複数端末への配布・GC時期の初期化）と、同期・リンクのトラブル対応は
**`references/setup.md`** を読むこと。「この環境に記憶システムをセットアップして」と
言われたら必ずそのファイルに従う。

## 型の背景

この型が生まれた経緯・診断フレームの詳細は、既定主体のワークスペース内の背景記録にある
（ハブのエージェントメモリの所在記録から辿れる）。手順の定義元はこのSKILL.md。
