# feedback-assetization — 暗黙の受け入れ基準の資産化

## 正本の所在

- **型（方法論の全文）**: `$HOME/Projects/Me/Workspace-Me/ideas/feedback-assetization.md`
- **L1/L2 ブリーフ正本ストア**: `$HOME/Projects/Me/Workspace-Me/ideas/creative-principles.md`
- **L0（案件ブリーフ）**: 各案件リポジトリの `docs/design-brief.md` 等（案件ごとに定める）

このSKILL.mdは運用手順の要約。判断に迷ったら型の全文を読む。

## 原則（1分版）

テスト・仕様で正解を機械判定できる仕事はエージェントが自走できるが、正解が人間の
頭の中の暗黙判断にしかない仕事は、検証ループが毎回人間を経由する。このとき大量に
発生するフィードバックの実体は「修正指示」ではなく**遅延した要件定義**である。
対策はフィードバックを消費して捨てず、一次データとして恒久資産（ブリーフ・受け入れ
基準・テスト）へ焼き付けること。ゴールはフィードバックゼロではなく**ラウンド数の削減**。

## 階層モデル

抽出した原則は「どの階層まで一般化できるか」で焼き付け先を分ける。

| 階層 | 範囲 | 置き場所 |
|---|---|---|
| L2 主体 | 人・会社の価値観。全成果物と事業判断に適用 | L1/L2 正本ストア |
| L1 ドメイン | 活動種別（サイト制作・コピー等）の原則。同ドメインの別案件へ転用可 | L1/L2 正本ストア |
| L0 案件 | この案件固有の決定。転用しない | 案件リポジトリのブリーフ |

- **昇格テスト**: 「案件（またはドメイン）が変わってもこの原則は生きるか？」——生きるなら1つ上の階層へ。
- **正本と複製**: 正本は各階層に1箇所。案件側ブリーフが自己完結を要する場合（ワークスペースを読めない環境のセッション等）はL1/L2を複製してよいが、複製側冒頭に正本の所在を明記し、改訂は必ず正本→複製の一方向。複製側だけを直したくなったら正本を先に直すサイン。
- 反例が出た原則は削除せず、適用条件を絞るか下位階層へ降格する。原則には根拠となった実例（どの案件のどの判断か）を添え、由来した案件数（n）を正直に保つ。

## 場面別の手順

### A. 制作系案件の開始時（事前採掘）

1. L1/L2 正本ストアを読み、この案件に関わる原則を作業コンテキストへ取り込む。
2. 暗黙基準の先出し: 形容詞（「洗練された」等）で語らせず、**参照例**（好きな実例・避けたい実例）をユーザーから集める。
3. 判断は「選ぶ」方が「語る」より一桁安い——**複数バリアント提示→選択**を既定の進め方にする。
4. L0 ブリーフの置き場所（例: `docs/design-brief.md`）と、フィードバック記録の規約（下記C）を最初に決めて案件のAGENTS.md等に書いておく。

### B. 制作系タスクの委譲時（同梱チェック）

委譲プロンプトに次を含めたか確認してから送る:

- L1/L2 正本ストアの該当原則（またはファイルを読む指示。委譲先がワークスペースを読めない場合は本文へ転記）。
- 案件にL0ブリーフがあればその参照。
- 「ユーザーのフィードバックを受けたら、判断理由をdocsに逐次注記する」指示（下記C）。

### C. 作業中（フィードバックの記録）

ユーザーのフィードバックで判断が変わるたび、案件の設計ドキュメントに
**「(日付 ユーザー指摘/判断) before → after ＋理由」** を逐次注記する。
git のコミットメッセージにも何を差し替えたかが残る粒度でコミットする。
この記録が後の抽出の一次データになる——トランスクリプトは消えるが docs と git 履歴は残る。

### D. 抽出（ブリーフを起こす・更新する）

フェーズ完了時、フィードバック多発の後、またはユーザーの依頼で実行する。

1. **ソース収集**: 案件の git 履歴（差し替え系コミットのdiff）・設計docsの判断注記・PR/Issueコメント・残っていればセッショントランスクリプト。長大なログの調査は、ハーネスがサブエージェントを持つならそちらへ委譲する。
2. **インベントリ化**: 各フィードバックを before/after・カテゴリ・背後の好み（推測は推測と明示）で一覧化する。
3. **パターン要約**: 繰り返し現れる好みの軸を5〜10個に要約する。頻出ほど重要。
4. **階層判定**: 各パターンに昇格テストを適用し、L0/L1/L2 に仕分ける。
5. **焼き付け**: L0→案件ブリーフへ、L1/L2→正本ストアへ追記（実例つき）。案件側に複製がある場合は正本→複製の順で更新する。機械判定できる基準（レイアウト崩れ・コンソールエラー・受け入れ条件等）はブリーフでなくテスト・ハーネス側へ落とす。
6. **不足の明示**: ブリーフ化に足りない領域（ユーザーに聞くべきこと）を成果物の末尾に列挙する。

### E. 減衰測定

次のイテレーションではブリーフを同梱してから着手し、フィードバックのラウンド数が
前回より減ったかを観察する。減っていなければ、ブリーフの記述が抽象的すぎる
（実例が足りない）か、階層判定が誤っている可能性を疑う。

## 診断への使い方

「マイクロマネジメントになっている」という相談が出たら: その仕事の
①判定の外部化度（機械判定できる正解があるか）と②仕様の圧縮率（言葉で伝えきれるか）
を確認する。両方低ければそれは仕事の性質であり、処方はこのスキルのサイクル
（記録→抽出→焼き付け→同梱）の導入。エージェントやユーザーの側の欠陥として
扱わない。
