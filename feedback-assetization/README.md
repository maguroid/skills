# feedback-assetization

「暗黙の受け入れ基準の資産化」の型と、その基盤であるエージェント中立の記憶システムを
一体で提供するスキル。

## 何を解決するか

テストや仕様で正解を機械判定できる仕事はエージェントが自走できるが、正解が人間の
頭の中の暗黙判断にしかない仕事（デザイン・コピー・文書・コードの品質基準や
レビュー観点・運用・調査・事業判断）では、検証ループが毎回人間を経由し、
マイクロマネジメントのようなフィードバック往復が大量発生する。

このスキルは、そのフィードバックを「遅延した要件定義」とみなして記録・抽出し、
スコープ階層で仕分けて恒久資産へ焼き付ける:

- **L0 案件** — この案件固有の決定。案件リポジトリのブリーフへ。
- **L1 ドメイン** — 活動種別（サイト制作・コピー等）の原則。別案件へ転用可。
- **L2 主体** — 人・会社の価値観。全成果物と事業判断に適用。

昇格テスト（「案件が変わっても生きるか？」）で階層を判定し、機械判定できる基準は
ブリーフでなくテスト・ハーネスへ落とす。ゴールはフィードバックゼロではなく
ラウンド数の削減。

## 記憶システム（基盤）

型の運搬基盤として、エージェント中立の「ハブ型」メモリを内蔵する。

- **保存**: ハブワークスペースのリポジトリ内 `agent-memory/`（1ファイル1知識の
  Markdown＋索引 `MEMORY.md`）。どのハーネス・ツールからも読み書きできる。
- **同期**: ハブリポジトリのgit（既存フック）に乗せる。専用機構なし。
- **接続**: Claude Code はシンボリックリンクで自動メモリ機能（自動注入・自動書き込み）
  に、codex 等はハブの AGENTS.md の読み込み指示で接続。
- **衛生**: 定期GC（重複統合・正本への圧縮・ハブ跨ぎ知識の昇格判定）。時期は
  `MEMORY.md` 冒頭の「最終GC」日付による自己記述方式で、タスクシステム不要。
- **境界**: ハブ境界＝信頼境界。知識は自動ではハブを跨がず、GCの昇格判定という
  意図的なゲートだけを通ってグローバル資産（CLAUDE.md・スキル・wiki）へ昇格する。

## 導入方法

スキルは「一度も呼ばれる前」には動けないため、インストール後に最初の一声が必要。

1. このスキルをディスカバリディレクトリ（`$HOME/.agents/skills` と
   `$HOME/.claude/skills`）にリンクする（リポジトリの `bootstrap.sh` で自動）。
2. エージェントに **「この環境に記憶システムをセットアップして」** と伝える。
   ハブの指定（または新設）→ ストア作成 → 同期フック設置 → ハーネスアダプタ接続 →
   グローバル指示ファイルへの常時ルール追記（要承認）→ GC時期の初期化まで、
   スキル内の `references/setup.md` がガイドする。
3. 以降は自動運転: タスクの開始・委譲時にブリーフが同梱され、フィードバックが
   記録・抽出・昇格され、「最終GC」日付が30日を超えるとセッションがGCを提案する。

## 使い方の入口

- 新しい案件・取り組みの開始時（事前採掘: 参照例の収集、複数バリアント提示の既定化）
- タスクの委譲時（ブリーフ同梱チェック）
- 「ブリーフ抽出して」「フィードバックを資産化して」（抽出の実行）
- フェーズ完了時・フィードバック多発後（抽出の提案）
- 「メモリ整理して」／「最終GC」日付の30日超過（定期GC）
- 「マイクロマネジメントになっている」という相談（診断と処方）
