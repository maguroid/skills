# セットアップ・導入・トラブルシュート

`feedback-assetization` スキルのセットアップ手順の定義元。
「この環境に記憶システムをセットアップして」への応答はこのファイルに従う。

## 1. 常時層の設置（資産化ルールの二重化）

最頻出の入口（タスク委譲時のブリーフ同梱）はスキルのトリガー任せにせず**常時層**で
二重化する。ハーネスのグローバル指示ファイル（Claude Code なら `~/.claude/CLAUDE.md`、
codex ならグローバル AGENTS.md 等）へ、次の趣旨のルール追記を提案する:

> 受け入れ基準が人の頭の中にしかない仕事では、修正フィードバックを「遅延した
> 要件定義」という一次データとして扱い、消費して捨てない（定義元:
> `feedback-assetization` スキル）。タスクの開始・委譲時は、主体のL1/L2ブリーフ
> 正本の該当原則を同梱する（所在はエージェントメモリで解決し、なければユーザーに
> 確認して記録する）。フィードバックで判断が変わったら before/after と理由を
> 案件の設計ドキュメントに逐次注記する。フェーズ完了時やフィードバック多発後は
> ブリーフ抽出を提案する。

グローバル指示ファイルは環境の持ち主の資産なので、**必ず承認を得てから**書き込む。

## 2. 記憶システムの導入（ハブごと・冪等）

1. **ハブとストアの用意**: ハブが無い環境では、中心となるワークスペースリポジトリを
   ハブに指定（または新設）する。`<ハブ>/agent-memory/` を作る。メモリがまだ無ければ、
   冒頭に「最終GC: <当日>」行だけを持つ `MEMORY.md` と空ディレクトリで開始する。
   ハーネスのネイティブメモリ（Claude Code の auto-memory 等）に引き継ぎたい記録が
   あれば、選別して**コピー**する（移設・リンクはしない。ネイティブ層とはストアを
   共有しない——SKILL.md「ハーネス接続」の分離方針が定義元）。
2. **同期フックの確認・設置**: ハブに自動同期（プル・コミット・プッシュ）の仕組みが
   あるか確認し、無ければ設置を提案する。Claude Code なら `.claude/settings.json` の
   フックで SessionStart に `git pull --ff-only`、Stop に変更があれば
   `git add -A && git commit && git push` を仕込む構成が実績ある。他ハーネスでは
   同等のフック機構を使うか、手動同期ならその旨をハブ文書に明記する。
3. **ハーネス接続フック**: このスキル同梱の `scripts/memory_inject.sh`（索引注入）と
   `scripts/dreaming_check.sh`（dreaming 候補の検知）を、各ハーネスのグローバル
   SessionStart フックに登録する——Claude Code は `~/.claude/settings.json` の
   `hooks.SessionStart`、codex は `~/.codex/config.toml` の `[[hooks.SessionStart]]`。
   両スクリプトは `~/.agents/hubs.md` からハブを解決するので、ハブ追加時は
   レジストリ追記だけで効く。フック設定は dotfiles 配布機構（chezmoi 等）に載せて
   全端末へ配布する。グローバル設定への書き込みは**必ず承認を得てから**行う。
4. **フック未対応ハーネスの代替**: ハブの AGENTS.md に「作業開始時に
   `agent-memory/MEMORY.md` を読む」指示を含める（次項のポインタ節に含めてよい）。
5. **リンタ除外**: ハブに文書規約チェック（OKF 等）があれば `agent-memory/` を除外に追加。
6. **ハブ文書へのポインタ追記**: ハブの AGENTS.md（または CLAUDE.md）に薄い節を1つ
   追加する。書くのは「実体の場所と接続方式」「同期の乗り先（フック）」「運用の定義元は
   `feedback-assetization` スキルであること」「ハブ固有の除外設定」のみ。
   **GC手順や規約の本文をハブ側に複製しない**（ドリフトの温床になる）。
7. **ハブレジストリへの登録**（複数ハブ運用時）: グローバル層のレジストリ
   （既定 `$HOME/.agents/hubs.md`。無ければ新設し、配布機構があればそれに載せる）へ、
   このハブのパス・主体・扱う知識の範囲・機密レベルを1エントリ追記する。
   clone 用に `- リモート:`（リポジトリの SSH URL）と、リモート名が origin 以外の
   場合のみ `- リモート名:` も書く。レジストリの
   位置づけ（読み取りゲート、引く3場面）は SKILL.md の「記憶システム」節が定義元。
   Claude Code アダプタの配布スクリプトのハブ一覧（機械層）とレジストリ（意味層）は
   役割が異なるため、ハブの追加・削除では両方を同時に更新する。
8. **索引更新**: ハブにルート索引（`index.md` 等）があれば `agent-memory/` の一行を追加。
9. **GC時期の初期化**: `MEMORY.md` 冒頭に「最終GC: <当日>」行が無ければ追加する。
10. **検証**: ハブ cwd で新規セッションを開始して索引が注入されること、ハブの
    コミット・プッシュでメモリが同期されること、リンタが通ることを確認する。

## 3. トラブルシュート

- **新端末でメモリが見えない** → ①ハブリポジトリのクローン ②skills リポジトリの
  bootstrap（フックが呼ぶスクリプトの実体）③dotfiles の適用（フック設定の配布）の
  3点が揃っているか確認する。
- **セッション開始時に索引が注入されない** → cwd がハブ配下か（`~/.agents/hubs.md` の
  パス宣言と一致するか）、フック設定が生きているか（Claude Code は `/hooks` で確認）、
  スクリプトを直接実行して出力があるかの順で切り分ける。
- **ハブのリポジトリ改名・移動** → `~/.agents/hubs.md` のパス宣言を更新する
  （フックは cwd の前方一致で解決するため、レジストリ更新だけで追随する）。
- **旧シンボリックリンク運用（2026-07-05 以前）からの移行** →
  `~/.claude/projects/<プロジェクトディレクトリ名>/memory` がハブの `agent-memory/` への
  リンクになっていたら削除し、実ディレクトリを作り直す（ネイティブ層は空から開始）。
  ハブ側の実体はそのまま正本として残る。
