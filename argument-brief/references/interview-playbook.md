# Interview Playbook

一問一答の質問候補・分岐・深掘り方法。**1ターン1質問**を厳守する。ここにある質問群は「並べて出す台本」ではなく「次の1問を選ぶための引き出し」である。既知の情報（会話・引数・メモ・下書き・既存資料・過去回答）から分かる項目は飛ばす。

## 基本フロー（順序は固定しない）

回答済み・既知の項目は飛ばし、未確定で最も情報価値の高いものを次に問う。

1. Audience → 2. Decision → 3. Core claim → 4. Opposing view / Our response → 5. Non-goals → 6. Why now → 7. Evidence needed → 8. Retraction condition

ただし、ユーザーの最初の入力が既にどれかを埋めているなら、そこを仮置きして空欄から始める。

## 項目別・質問候補と深掘り

### Audience
- 主質問：「この資料は、誰の、どの判断のために書きますか？」
- 詰まったら選択肢：意思決定者本人 / 決裁ラインの上長 / 投資委員会・審査会 / 顧客の担当者 / 社内の実行部隊
- 深掘り：その人の役職・関心・今置かれている状況（予算/時期/プレッシャー）を1つだけ問う。
- NG：「経営層」だけで止めない。誰の何の決定かまで具体化する。

### Decision
- 主質問：「読み終えた相手に、何を“決めて”ほしいですか？（承認/中止/開始/変更/優先順位付けのどれか）」
- 深掘り：動詞で言わせる。「理解してほしい」「知ってほしい」は Decision ではない → 「理解した上で何をしてほしいか」を問う。
- NG：状態（〜を理解する）で止めない。行動に落とす。

### Core claim（最重要）
- 主質問：「相手に、最終的に何を“信じて”ほしいですか？ 一文で言い切ると？」
- テーマが返ってきたら（AI活用/生産性向上/組織変革/顧客理解/データ活用/営業力強化 等）：そのままにせず、次のうち1つを問うて主張化する。
  - 「何と比較して、それを選ぶ？」（比較）
  - 「何を先にやり、何を後回しにする？」（優先順位）
  - 「何を始め、何をやめる？」（始める/やめる）
  - 「それは何を引き起こす、という因果？」（因果）
  - 「どの条件で成り立ち、どの条件で成り立たない？」（適用条件）
  - 「世間の一般的な見方は何で、それのどこが違うと言いたい？」（反論）
- 具体化の軸（必要に応じ1つずつ）：誰が / 何を / 何と比較して / どの状況で / 何のために / 何を始める / 何をやめる / 何を優先する。
- 整形：ユーザーの言葉を使い、原則1文・反論可能にする。AIが新しい主張を発明しない。

### 抽象語の深掘り
回答が「重要/必要/強化/推進/活用/最適化/高度化/価値向上/生産性向上/顧客中心」等だけの時：
- 「それが“できている状態”と“できていない状態”は、具体的に何が違いますか？」
- 「いま誰が、何を、どうしているのを、どう変えるという話ですか？」
- 事実は捏造しない。ユーザーが事実を出せないなら Hypothesis として印を付ける。

### 「AもBも」の切り分け
複数論点が同列に並んだ時、次から1つ問う：
- 「どちらを先にやりますか？」（順序）
- 「どちらを優先しますか？」（優先）
- 「どの条件で使い分けますか？」（条件）
- 「今回の資料ではどちらを“扱わない”ことにしますか？」（対象外）
- 「どちらが目的で、どちらが手段ですか？」（目的/手段）
- 根拠なき二者択一は作らない。順序・対象・条件・目的手段として整理する。

### Opposing view（藁人形にしない）
- 主質問：「あなたの主張を“採用しない”と判断する、賢くて合理的な人は、何を根拠にそう言いますか？」
- 詰まったら、最強の反対候補を2〜3個提示し「実際の反対に一番近いのはどれ？」と確認。候補は、コスト/リスク/タイミング/代替案/前例/測定不能性 などの観点から作る。
- Our response：選ばれた Opposing view に**直接**答える。論点をずらさない。

### Non-goals
- 主質問：「この資料で“扱わない”こと、結論を広げない範囲、今回は決めないことは何ですか？」
- 補助：「誤解されたくない主張は？」「今回優先しないことは？」
- 目的：スコープを締め、Core claim のエッジを守る。

### Why now
- 主質問：「なぜ“今”この判断が必要なのですか？ 半年後ではダメな理由は？」
- 深掘り：期限・機会・コスト増・競合・既に起きている変化 のどれか。

### Evidence needed（Fact/Inference/Hypothesis を分ける）
- 「いま手元にある根拠（Existing）は？」
- 「主張を支えるために、追加で取りに行く根拠（Additional）は？」
- 「主張に不利な、反対証拠（Counter-evidence）になりうるものは？」
- 「証明が難しいが前提にしていること（Assumptions to validate）は？」
- 受け取った各情報に Fact / Inference / Hypothesis のラベルを内部で付ける。未確認の事実をAIが埋めない。

### Retraction condition（必ず聞く）
- 主質問：「どんな“事実”が分かったら、この主張を撤回または修正しますか？」
- 曖昧回答（「成果が出なければ」「状況が変われば」「うまくいかなければ」）は受け取らず、観測可能化する：
  - 対象（誰/何の指標）/ 指標 / 比較対象 / 期間 / 閾値 / 期待した因果が成立しない状態
  - 例の型：「対象◯◯で、指標△△が、比較対象□□に対して、期間◇◇のうちに、閾値◯%を超えなければ撤回」
- 思いつかない場合、Core claim に対応した候補を2〜3個提示して選んでもらう。

## 「反証して」への手順
1. 現在の Core claim を1文で再掲。
2. 次を示す：成立しない条件 / 最強の反対意見 / 証拠の弱い箇所 / 対象が広すぎる箇所 / 主張を維持したまま精度を上げる案。
3. 「維持する / 適用範囲を限定する / 主張を変更する」を選んでもらう。
4. 選択に応じて Core claim を更新（変更はユーザー決定。AIは整形のみ）。

## ターンの型（毎回）
1. 言い換え（1〜3文）。2. 必要なら解釈確認。3. 次の1問。— Brief 全体は再掲しない。2〜4問ごとに短い途中確認を任意で挟む。
